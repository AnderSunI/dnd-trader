# D&D Trader Operational Memory

## 0. Актуальный источник состояния
- Для продолжения работы актуальный источник — `SESSION_MEMORY.md`.
- `AGENTS.md` хранит общий operational background и правила проекта, но не обязан отражать последнюю точку остановки.
- Если пользователь пишет `чекни мемори`, `мемори файл`, `session memory` или спрашивает, где остановились:
  1. сначала читать `SESSION_MEMORY.md`;
  2. потом проверять ветку и `git status`;
  3. только после этого сверяться с `AGENTS.md`, если нужны общие правила.
- После каждого существенного прохода обновлять `SESSION_MEMORY.md`, даже если изменённые рабочие файлы ещё не коммитятся.

## 1. Что это за проект
- Проект: `D&D Trader`.
- Это живой проект, не учебная болванка и не пустой pet-project.
- Продуктовая идея: `D&D / BG3-inspired trader + cabinet + GM tool + LSS + encyclopedia + party systems`.
- Главный принцип: не выдумывать архитектуру мимо существующего кода.
- Нельзя ломать уже работающие модули ради красоты, абстрактной чистоты или “идеального” рефактора.

## 2. Как работать по проекту
- Сначала анализ текущего состояния runtime и файлов.
- Потом короткий план.
- Потом код.
- По возможности делать один узкий шаг за раз.
- Не коммитить и не пушить без отдельного подтверждения.
- Не создавать новые ветки без явной нужды и запроса.
- Не делать “генеральную уборку” без запроса.
- Не делать foundation-рефакторы без usable результата.
- Если меняется файл, по запросу быть готовым показать полный итоговый файл.
- Перед большими правками заранее обозначать:
  - какие файлы меняются
  - зачем именно они
  - где риск сломать текущий flow

## 3. Ветки и git-правила
- `main` = прод / Render.
- `develop` = основная рабочая база пользователя.
- `work` = вспомогательная рабочая ветка для другого человека.
- Если текущая сессия идёт в другой ветке, сначала явно это проверить и озвучить.
- Перед началом работы всегда:
  - определить текущую ветку
  - проверить `git status`
  - не начинать кодить, пока не понятен контекст ветки
- Не пушить без отдельного подтверждения.
- Не делать опасные git-действия без явного запроса.
- Не создавать ветки “на автомате”.

## 4. Пользователь и стиль общения
- Пользователь технарь, но не классический программист.
- Лучше воспринимает общую картину, связь блоков и практический смысл.
- Не любит воду, повторение, бессмысленные прыжки между подходами и ветками.
- Говорить прямо, по делу, без лишней академичности.
- Если что-то не получилось, говорить честно.
- Не выдавать “похоже” за “сделано точно”.
- Если есть сильный разрыв между референсом и текущим результатом, прямо объяснять почему.

## 6. Очень важный блок по предметам и данным
- Ничего не терять.
- Лучше сохранить грязные данные, чем потерять механику.
- Нельзя механически вырезать `spells`, `states`, `mechanics pages` как мусор.
- Сохранять:
  - `lore`
  - `mechanics_text`
  - `grants`
  - `source`
  - useful notes
- Не схлопывать насильно `BG3 / PHB / dnd.su` в один “идеальный предмет”.
- Держать layered pipeline:
  - `raw -> preserved/category master -> overrides -> global master -> canonical`

## 7. UI / визуал: постоянный контекст
- Нужен `premium dark fantasy RPG UI`.
- Не `SaaS`.
- Не `generic dashboard`.
- Не просто “тёмная тема”.
- Ориентир по ощущению: `BG3-inspired`, но не копия 1 в 1.
- Игровой режим важнее админского ощущения.
- Общий стиль:
  - deep blue-black base
  - premium dark glass panels
  - pale cyan buttons
  - muted brass/gold accents
  - soft glow
  - rounded framed panels
  - readable hierarchy
- `Мой аккаунт` должен ощущаться как игровой профиль / hub.
- `Master Room` должен ощущаться как отдельный `GM mode / control room`.
- `Trader UI` должен быть в том же визуальном каноне.
- `Combat UI` должен быть game-like:
  - initiative strip
  - combat log
  - dice log

## 8. Последний важный вывод по референсу `Общий.png`
- Причина, почему текущий результат всё ещё далеко от канона, не в одном-двух цветах.
- Уже совпало:
  - тёмный `navy/black` базовый тон
  - `cyan/brass` accents
  - framed panels
  - более тяжёлые кнопки и pills
  - initiative strip
  - tactical stack
  - `Account / Master Room / Battle / Trader` уже не выглядят как голый светлый CRUD
- Почему всё ещё далеко:
  - текущий UI всё ещё живёт на старой кабинетной разметке, а референс построен как цельная дизайн-система с жёстким `layout rhythm`
  - в `frontend/js/account.js` и `frontend/js/cabinet.js` всё ещё очень много inline-стилей
  - нет настоящего системного shell как в референсе:
    - тяжёлая левая rail
    - единая верхняя иерархия
    - одинаковый ритм секций
    - единое пространство отступов / радиусов / высот
  - типографика не совпадает: в референсе она более собранная и `editorial/fantasy-premium`
  - материалы не дожаты: панели в референсе многослойные и ощущаются как цельный UI-material
  - композиция не совпадает 1 в 1:
    - `Account` всё ещё местами читается как профиль + формы
    - `Master Room` всё ещё местами читается как кабинет с GM-секциями
    - `Trader` всё ещё живёт своим мини-языком
  - фон, декоративные линии, угловые акценты, рамочные рёбра и крупные композиционные массы пока не воспроизведены
  - компоненты ещё не унифицированы полностью:
    - `.cabinet-block`
    - `.stat-box`
    - `.trader-card`
    - `.tab-bar`
    - `.meta-item`
    - `.btn`
- Самая точная формулировка:
  - сейчас это `design skin over old runtime`
  - а в референсе нужен `system-first UI`
- Что особенно мешает в коде:
  - очень много inline-стилей в:
    - `frontend/js/account.js`
    - `frontend/js/cabinet.js`
    - частично `frontend/js/render.js`
  - из-за этого даже хорошие CSS-классы не могут полностью взять UI под контроль
  - нельзя лечить модуль за модулем бесконечно; сначала нужна общая система, потом пересадка экранов

## 9. Роадмап по визуальному сближению с `Общий.png`
1. Сначала собрать каноническую design-system основу в `frontend/css/styles.css`.
- Единые токены:
  - panel depths
  - border language
  - glows
  - radii
  - header scales
  - chip sizes
  - button heights
  - spacing scale
- Отдельные классы для:
  - shell
  - rail
  - hero
  - framed section
  - tactical card
  - segmented tabs
  - economy blocks
- Почему первым: пока этого нет, каждый экран будет только “примерно похож”.

2. Потом убрать критичную массу inline-стилей из `frontend/js/account.js` и `frontend/js/cabinet.js`.
- Не весь рефактор подряд.
- Только визуальные inline-блоки, которые ломают ритм.
- Почему: сейчас именно они не дают попасть в единый канон.

3. Потом пересобрать shell кабинета и rail-навигацию.
- Файлы:
  - `frontend/js/cabinet.js`
  - `frontend/css/styles.css`
- Цель:
  - левая rail как в референсе
  - общий верхний режимный header
  - consistent content frame
- Почему: без этого `Account` и `Master Room` всегда будут ощущаться как вариации одной вкладки.

4. Потом отдельно дожать `Account Hub`.
- Файл: `frontend/js/account.js`
- Цель:
  - hero
  - segmented nav
  - featured rail
  - framed profile fields
- Формы должны стать встроенной частью панели, а не самостоятельным видом страницы.

5. Потом отдельно дожать `Master Room`.
- Файл: `frontend/js/cabinet.js`
- Нужен более жёсткий `GM workspace rhythm`:
  - table rail
  - command block
  - player roster
  - access block
  - session / event area
- Сейчас уже лучше, но это ещё не полноценный `GM control panel`.

6. Потом довести `Trader UI` до общего канона.
- Файл: `frontend/js/render.js`
- Сейчас это самый выбивающийся слой после shell.
- Нужно:
  - одна и та же панельная анатомия
  - те же chips
  - те же headers
  - те же economy blocks
  - меньше ощущения “каталога”

7. Потом финальный polish `Battle`.
- Уже ближе всех к нужному настроению.
- Осталось:
  - точнее типографика
  - тоньше separators
  - строже event hierarchy
  - более каноничный right tactical column

## 16. Актуальное уточнение от 2026-05-02
- LSS как отдельный модуль пока не трогать.
- Главная визуальная цель по `Бою`: максимально близкое / почти 1-в-1 попадание в `frontend/static/images/Referens/Бой.png`.
- `Master Room` нужно улучшать без потери функциональности; референс `Мастеррум.png` использовать осторожно, потому что часть решений там сгенерирована с отсебятиной.
- Текущий правильный подход:
  1. сначала обкатать визуальный приём на небольших модулях личного кабинета;
  2. показать пользователю;
  3. после подтверждения переносить этот rhythm/material/shell на следующие модули;
  4. в конце применять к самым сложным экранам — `Master Room` и `Бой`.
- Текущий малый модуль для обкатки: `Чат` в личном кабинете.
- Важные текущие проблемы:
  - при входе в личный кабинет не должно быть ощущения маленького модального окна;
  - элементы не должны быть чрезмерно крупными;
  - экран должен ощущаться почти fullscreen, как в референсах;
  - сайт не должен подвисать из-за eager-загрузки лишних кабинетных модулей;
  - левый блок `Master Room` в будущем должен быть кратко сжат по умолчанию и раскрываться по кнопке;
  - текущий `Master Room` считается нечитаемым и требует отдельного прохода после проверки малого модуля.

## 17. Старт следующей сессии без вводных
Если новая сессия начинается без подробного объяснения пользователя, считать актуальным именно этот порядок:
1. Прочитать `SESSION_MEMORY.md`, особенно блок `CURRENT HANDOFF`.
2. Проверить:
   - `git branch --show-current`;
   - `git status --short`.
3. Кратко сообщить пользователю:
   - текущую ветку;
   - изменённые файлы;
   - что фокус сейчас `личный кабинет / чат`, а не LSS, Battle или большой Master Room.
4. Не начинать кодить до короткого плана.
5. Первый рабочий шаг:
   - восстановить/проверить локальный runtime;
   - открыть авторизованный личный кабинет;
   - перейти в `Чат`;
   - снять screenshot `1672x941`;
   - сравнить с `frontend/static/images/Referens/чат.png`;
   - сделать один узкий visual pass по fullscreen/density/messenger layout.
6. После правки проверить:
   - `node --check frontend/js/app.js`;
   - `node --check frontend/js/account.js`;
   - `node --check frontend/js/cabinet.js`;
   - `git diff --check -- frontend/js/app.js frontend/js/account.js frontend/js/cabinet.js frontend/css/styles.css`.
7. Обновить `SESSION_MEMORY.md`.

Текущие ориентировочные проценты близости:
- `Личный кабинет shell/fullscreen`: ~60–65%.
- `Чат как тестовый малый модуль`: ~50–55%.
- `Master Room`: ~25–30%.
- `Бой`: ~40–45%, но целевое требование пользователя — почти `1 в 1`.

Текущие рабочие файлы незакоммиченного прохода:
- `frontend/css/styles.css`
- `frontend/js/account.js`
- `frontend/js/app.js`
- `frontend/js/cabinet.js`
- `AGENTS.md`

Нельзя без нового прямого запроса:
- трогать LSS;
- начинать финальный pass по `Бою`;
- делать широкий rewrite `Master Room`;
- продолжать старый trader-modal план из прошлой memory, пока пользователь явно не вернёт эту задачу.
