# Правила модульности (чтобы не «на каждый чих»)

Короткий ответ: **модулизация нужна по доменам и ответственности**, а не по одной маленькой функции.

## Когда НЕ создаём новый модуль

Не выносим код в отдельный файл, если:
1. это 1–2 короткие функции, которые используются только в одном месте;
2. логика тесно связана с локальным состоянием текущего файла и не переиспользуется;
3. вынос добавляет больше «прокидывания зависимостей», чем пользы;
4. изменение разовое и не ожидается развитие этой зоны.

В таких случаях оставляем код рядом, но:
- добавляем комментарий «зачем так»;
- держим компактные функции;
- избегаем дублирования.

## Когда создаём модуль

Выносим в модуль, если есть хотя бы **2 из 4** признаков:
1. есть отдельная предметная зона (например: `trader`, `inventory`, `auth_profile`);
2. блок вырос и мешает читать сценарий оркестратора (`app.js`);
3. код уже переиспользуется или явно будет переиспользоваться;
4. внутри блока есть собственный мини-поток (валидация → действие → обновление UI/состояния).

## Практический порог (эвристика)

- Если блок в `app.js` > ~80–120 строк и имеет свою тему — кандидат на вынос.
- Если после выноса модуль < ~30 строк и без перспектив роста — лучше не дробить.
- Предпочтительно: **один модуль = один связный сценарий/домен**, а не «модуль на каждую helper-функцию».

## Целевая структура для фронта

- `frontend/js/modules/trader/` — всё про трейдера (open/restock/filters/ui sync).
- `frontend/js/modules/inventory/` — cart/reserved/inventory/checkout.
- `frontend/js/modules/auth_profile/` — auth/profile modal flow.

Внутри домена допустимы helper-файлы, но только если они реально улучшают читаемость и тестируемость.

## Правило перед каждым новым файлом

Перед созданием нового модуля отвечаем на 3 вопроса:
1. Какую **одну ответственность** он берёт?
2. Что станет **проще читать** в `app.js` после выноса?
3. Почему это не лучше оставить рядом?

Если на любой вопрос нет ясного ответа — модуль пока не создаём.
